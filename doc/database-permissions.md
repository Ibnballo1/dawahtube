# Database permission setup

## audit_logs table — INSERT only

The application database user must be granted INSERT-only on audit_logs.
Run once on the Supabase SQL editor (not in migrations — this is infra setup):

```sql
-- Replace 'app_user' with your actual Supabase DB role name
-- Found in: Supabase Dashboard → Settings → Database → Connection string
REVOKE UPDATE, DELETE ON audit_logs FROM app_user;
GRANT INSERT, SELECT ON audit_logs TO app_user;
```

This is enforced at the Postgres permission level, independent of the
application. Even if application code attempted a DELETE, Postgres would
reject it with a permissions error.

## Verification

```sql
SELECT grantee, privilege_type
FROM information_schema.role_table_grants
WHERE table_name = 'audit_logs';
```

Expected output: INSERT and SELECT for app_user. No UPDATE. No DELETE.

Note on where to find your Supabase DB role: Supabase Dashboard → Settings → Database → Connection info → User field. It is typically postgres for the owner role or a custom role if you create one. The SQL in docs/database-permissions.md is run once manually in the Supabase SQL editor — it is intentionally not a Drizzle migration because it is a one-time infrastructure configuration, not a schema change.

Add to migration file

-- src/core/database/migrations/0002_search_vectors.sql
-- (This file is generated by drizzle-kit — the raw SQL shown here is
-- for reference. You add it via a custom migration, not by hand.)

-- Use 'simple' config for title (no stemming — preserves scholar names, Arabic transliterations)
-- Use 'english' for description/transcript (stemming helps recall on body text)
-- The language of the content is stored in lectures.default_language.
-- A future migration can make this dynamic per-row; for MVP, per-column is sufficient.

CREATE OR REPLACE FUNCTION update_lecture_search_vector()
RETURNS TRIGGER AS $$
BEGIN
NEW.search_vector :=
setweight(to_tsvector('simple', coalesce(NEW.title, '')), 'A') ||
setweight(to_tsvector('english', coalesce(NEW.description, '')), 'B') ||
setweight(to_tsvector('english', coalesce(NEW.transcript, '')), 'C');
RETURN NEW;
END;

$$
LANGUAGE plpgsql;

CREATE OR REPLACE FUNCTION update_article_search_vector()
RETURNS TRIGGER AS
$$

BEGIN
NEW.search_vector :=
setweight(to_tsvector('simple', coalesce(NEW.title, '')), 'A') ||
setweight(to_tsvector('english', coalesce(NEW.excerpt, '')), 'B') ||
setweight(to_tsvector('english', coalesce(NEW.content, '')), 'C');
RETURN NEW;
END;

$$
LANGUAGE plpgsql;

CREATE OR REPLACE FUNCTION update_book_search_vector()
RETURNS TRIGGER AS
$$

BEGIN
NEW.search_vector :=
setweight(to_tsvector('simple', coalesce(NEW.title, '')), 'A') ||
setweight(to_tsvector('simple', coalesce(NEW.author_name, '')), 'A') ||
setweight(to_tsvector('english', coalesce(NEW.description, '')), 'B');
RETURN NEW;
END;

$$
LANGUAGE plpgsql;

-- Attach triggers
CREATE TRIGGER lectures_search_vector_update
  BEFORE INSERT OR UPDATE ON lectures
  FOR EACH ROW EXECUTE FUNCTION update_lecture_search_vector();

CREATE TRIGGER articles_search_vector_update
  BEFORE INSERT OR UPDATE ON articles
  FOR EACH ROW EXECUTE FUNCTION update_article_search_vector();

CREATE TRIGGER books_search_vector_update
  BEFORE INSERT OR UPDATE ON books
  FOR EACH ROW EXECUTE FUNCTION update_book_search_vector();

-- GIN indexes on all three vectors
CREATE INDEX IF NOT EXISTS lectures_search_idx ON lectures USING GIN(search_vector);
CREATE INDEX IF NOT EXISTS articles_search_idx ON articles USING GIN(search_vector);
CREATE INDEX IF NOT EXISTS books_search_idx    ON books    USING GIN(search_vector);

# 1. After Phase 2 schema is written and all tables exist, run:
npm run db:generate
# Drizzle creates: src/core/database/migrations/0001_initial_schema.sql

# 2. Create the custom migration file manually:
# src/core/database/migrations/0002_search_vectors.sql
# Paste the SQL above into it.

# 3. Apply both:
npm run db:migrate
$$
